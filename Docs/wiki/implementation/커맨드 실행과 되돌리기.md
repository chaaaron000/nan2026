# 커맨드 실행과 되돌리기

> Sources: NaN2026 Command Undo/Clear 구현, 2026-07-22
> Raw: [Command Undo and Clear Implementation Snapshot](../../raw/implementation/2026-07-22-command-undo-clear-implementation.md)
> Updated: 2026-07-22

## Overview

물감통 사용은 `ICommand` 구현체로 캡슐화되고 `CommandController`가 성공한 실행 이력을 Stack으로 관리한다. `PaintBucketUseCommand`는 영향받은 셀의 실행 직전 상태만 저장하므로 마지막 행동 또는 모든 행동을 정확히 역순 복원할 수 있다. Undo와 Clear 버튼은 Unity Inspector에서 호출 가능한 public void 핸들러를 통해 각각 한 단계 되돌리기와 전체 되돌리기를 요청한다.

## 전체 실행 구조

```text
PaintBucketController.BucketUseRequested
    └─ GridTestController
         └─ PaintBucketUseCommand 생성
              └─ CommandController.Execute
                   ├─ command.Execute 실패 → 이력에 저장하지 않음
                   └─ command.Execute 성공 → undoStack.Push

UndoButton
    └─ HandleUndoButtonClicked
         └─ UndoLast
              └─ undoStack.Pop().Undo

ClearButton
    └─ HandleClearButtonClicked
         └─ UndoAll
              └─ 최근 커맨드부터 모두 Pop하고 Undo
```

`GridTestController`는 요청을 실제 커맨드로 조립하는 역할을 맡고, `CommandController`는 개별 물감 규칙을 모른 채 `ICommand`의 실행과 이력만 관리한다. `PaintBucketUseCommand`는 한 번의 물감통 사용에 필요한 모델·View·물감통 상태 변경과 복원을 소유한다.

## ICommand 계약

```csharp
public interface ICommand
{
    bool Execute();
    void Undo();
}
```

`Execute`의 bool은 커맨드가 실제로 실행되었는지를 나타낸다. 이미 실행된 같은 커맨드나 이미 소모된 물감통처럼 실행할 수 없는 경우 false를 반환하므로 Controller는 실패한 행동을 Undo 이력에 넣지 않는다. `Undo`는 성공했던 실행을 이전 상태로 복원한다.

현재 `Undo`에는 반환값이 없다. 따라서 Controller는 스택에 성공한 커맨드만 있다는 불변식에 의존하고, 커맨드가 자신의 중복 Undo를 방어한다.

## CommandController와 Stack

되돌리기는 행동의 반대 순서로 수행해야 한다. A 실행 후 B를 실행했다면 먼저 B를 복원해야 A 실행 직전 상태를 보존할 수 있으므로 후입선출 구조인 `Stack<ICommand>`가 맞다.

`CommandController`의 책임은 다음과 같다.

- null 커맨드 실행 방지
- `Execute`가 성공한 커맨드만 `undoStack`에 Push
- 마지막 성공 커맨드 하나를 Pop하고 Undo
- 모든 성공 커맨드를 최근 순서부터 Undo
- 새 플레이 상태로 교체할 때 이력만 폐기
- `CanUndo`를 통한 이력 존재 여부 제공

`ClearHistory`와 전체 되돌리기는 서로 다르다.

| 동작 | 커맨드 Undo | 격자 복원 | 물감통 복원 | 용도 |
|---|---|---|---|---|
| `UndoAll` | 수행 | 수행 | 수행 | 현재 퍼즐을 시작 상태로 되돌림 |
| `ClearHistory` | 수행하지 않음 | 변경하지 않음 | 변경하지 않음 | 기존 플레이 객체를 폐기할 때 참조 이력만 제거 |

새 테스트 격자를 생성할 때는 어차피 새 `GridState`, View와 물감통 목록으로 교체하므로 `UndoAll` 대신 `ClearHistory`를 호출한다. 이를 생략하면 스택의 커맨드가 이전 격자와 이전 물감통 ID를 계속 참조한다.

## PaintSnapshot을 이용한 부분 스냅샷

물감통 사용은 전체 격자가 아니라 확산에 포함된 셀만 변경한다. 따라서 `PaintBucketUseCommand`는 다음 값 타입을 List에 저장한다.

```text
PaintSnapshot
├─ Position: 영향받은 논리 좌표
└─ PaintState: 해당 좌표의 실행 직전 상태
```

전체 격자를 매 행동마다 복제하지 않으므로 저장 비용은 전체 셀 수가 아니라 실제 영향 셀 수에 비례한다. 또한 현재 사용 패턴은 좌표별 임의 검색 없이 실행 시 순차 추가하고 Undo 시 전체 순회하는 형태다. 이 때문에 해시 테이블과 버킷 관리 비용이 있는 Dictionary보다 `List<PaintSnapshot>`이 구조와 접근 패턴에 잘 맞는다.

확산 계산기가 같은 좌표를 중복 반환하지 않는 책임을 가진다는 전제가 있다. 향후 계산 결과의 중복을 커맨드 계층에서도 방어해야 하거나 좌표로 스냅샷을 자주 조회해야 한다면 Dictionary를 다시 고려할 수 있다.

## 정확한 PaintState 복원

`GridState.AddPaint`는 기존 상태에 비트 OR를 수행한다. 예를 들어 이미 `Red | Blue`인 셀에 Red를 적용했을 때 실행 결과만 보고 어떤 비트가 이번 행동에서 추가되었는지 안전하게 역산할 수 없다. Clear 물감도 기존의 임의 RGB 조합을 Empty로 바꾸므로 동일한 문제가 있다.

따라서 Undo는 반대 연산을 계산하지 않고 저장했던 값을 `GridState.SetPaint`로 그대로 덮어쓴다. 이후 같은 값을 `GridView.SetCellPaint`에 전달해 권위 있는 모델 상태와 화면 표현을 함께 동기화한다.

`PaintBucketUseCommand.Undo`의 순서는 다음과 같다.

1. 실행되지 않은 커맨드라면 종료한다.
2. 모든 `PaintSnapshot`의 좌표와 이전 상태를 모델에 복원한다.
3. 각 좌표의 View를 같은 상태로 갱신한다.
4. 사용했던 물감통 ID를 `PaintBucketController.Restore`로 복원한다.
5. 커맨드를 실행되지 않은 상태로 바꿔 중복 Undo를 방지한다.

## Unity 버튼 연결

Unity `Button.onClick`은 반환값과 매개변수가 없는 `UnityEvent`다. 로직 메서드인 `UndoLast`는 성공 여부를 bool로 반환하므로 Inspector의 정적 호출 후보로 사용할 수 없다. 현재 구현은 UI 어댑터 역할의 public void 메서드를 별도로 제공한다.

- `UndoButton` → `HandleUndoButtonClicked()` → 내부 `UndoLast()`
- `ClearButton` → `HandleClearButtonClicked()` → 내부 `UndoAll()`

현재 씬 직렬화에서도 두 버튼은 `CommandController` 컴포넌트의 해당 메서드에 연결되어 있다.

## Clear 버튼의 의미

현재 Clear는 단순히 모든 셀에 `GridState.ClearAll`을 호출하는 기능이 아니다. 지금까지 사용한 물감통 커맨드를 모두 역순으로 되돌려 다음 상태를 함께 복원한다.

- 영향받았던 셀의 원래 혼합색
- Clear 물감으로 지워졌던 셀의 원래 색
- 소모되었던 각 물감통의 사용 가능 상태
- Undo 스택의 빈 상태

따라서 UI 명칭이 Clear여도 도메인 의미는 “이번 시도의 모든 행동 취소” 또는 “초기 상태로 복원”에 가깝다. 향후 정말로 현재 색만 비우는 기능이 필요하다면 직접 `GridState.ClearAll`을 호출하기보다 별도의 `ClearGridCommand`로 만들어 그 행동 자체도 되돌릴 수 있게 하는 편이 일관적이다.

## 현재 한계와 확장 경계

- `CanUndo`는 제공하지만 버튼의 interactable 상태와 자동 동기화하지 않는다.
- Redo 이력과 재실행 UI는 없다.
- 실행 도중 예외가 발생해 일부 셀만 변경된 경우 자동 롤백하지 않는다.
- `Undo`가 void라서 물감통이나 View 복원 실패를 Controller가 감지할 계약은 없다.
- 애니메이션이 추가되면 커맨드 실행·Undo 중 입력 잠금과 완료 시점을 별도로 정의해야 한다.
