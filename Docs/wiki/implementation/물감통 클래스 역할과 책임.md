# 물감통 클래스 역할과 책임

> Sources: NaN2026 현재 구현, 2026-07-21
> Raw: [Paint Bucket Implementation Snapshot](../../raw/implementation/2026-07-21-paint-bucket-implementation.md)
> Updated: 2026-07-21

## Overview

현재 물감통 시스템은 스테이지에 저장되는 사양, 공통 시각 리소스, 오브젝트 생성, 개별 UI 표현, 인게임 상호작용 관리로 책임을 나눈다. `PaintBucket`은 색상과 범위만 가진 직렬화 데이터이고, `PaintBucketGenerator`가 이를 바탕으로 `PaintBucketView` 프리팹을 생성한다. `PaintBucketController`는 생성된 물감통을 개별 ID로 관리하고 `GridView`가 중계한 셀 클릭과 연결한다. 실제 확산과 셀 상태 변경은 아직 구현하지 않았으며 향후 커맨드 계층에 맡긴다.

## 전체 협력 구조

```text
스테이지 데이터 계층(예정)
    └─ IReadOnlyList<PaintBucket>
              │
              ▼
PaintBucketController.Initialize
    └─ PaintBucketGenerator.Generate
           ├─ PaintBucketVisualData.GetSprite
           └─ PaintBucketView 프리팹 생성 및 초기화

PaintBucketView 클릭
    └─ PaintBucketController가 BucketEntry 선택

CellView 클릭
    └─ GridView.CellClicked 중계
           └─ PaintBucketController.BucketUseRequested
                  └─ 물감 사용 커맨드 계층(예정)
```

초기화 호출자는 향후 스테이지 데이터를 소유하는 상위 관리자다. `PaintBucketController`는 `StageData` 전체를 직접 참조하지 않고 현재 스테이지의 물감통 목록만 전달받는다. 반면 물감통 공통 Sprite는 스테이지별 정보가 아니므로 `PaintBucketGenerator`가 `PaintBucketVisualData`를 직접 참조한다.

## PaintType

`PaintType`은 물감통이 수행할 행동 종류를 나타낸다.

| 값 | 의미 |
|---|---|
| `Red` | 셀 상태에 R을 추가할 물감 |
| `Green` | 셀 상태에 G를 추가할 물감 |
| `Blue` | 셀 상태에 B를 추가할 물감 |
| `Clear` | 도달한 셀의 RGB 상태를 모두 제거할 물감 |

`Clear`는 `PaintState.White`와 구분된다. White는 R, G, B가 모두 포함된 셀 상태이고, Clear는 색 상태로 남지 않는 삭제 행동이다.

## PaintBucket

`PaintBucket`은 물감통 한 개의 스테이지 사양을 나타내는 `[Serializable]` 일반 C# 클래스다.

- `PaintType`: 물감의 종류
- `Range`: 시작 셀을 포함한 확산 범위이며 최소값은 1

수량 필드는 두지 않는다. 예를 들어 `Red`, 범위 2 물감통이 두 개라면 스테이지 목록에 같은 사양의 `PaintBucket` 항목을 두 개 둔다. 각 항목은 별도의 View와 ID를 받아 독립적으로 선택·소모·복원된다.

`PaintBucket`은 원본 스테이지 데이터이므로 런타임에 소모 여부를 기록하지 않는다. 소모 상태는 Controller의 내부 관리 항목에 둔다.

## PaintBucketVisualData

`PaintBucketVisualData`는 물감통 종류별 공통 Sprite를 모아 둔 ScriptableObject다. `GetSprite(PaintType)` API가 R, G, B, Clear에 대응하는 Sprite를 반환한다.

이 데이터가 Generator에 있는 이유는 다음과 같다.

- 스테이지 관리자는 스테이지별 규칙 데이터만 다룬다.
- Sprite 교체는 VisualData 에셋 한 곳에서 수행한다.
- View는 Sprite가 어디에 저장됐는지 알지 않고 전달받은 결과만 표현한다.

향후 선택 또는 소모 전용 Sprite가 필요하면 상태를 인자로 받는 조회 API로 확장할 수 있다. 현재 선택 표현은 아직 구현하지 않았고, 소모는 오브젝트 비활성화로 처리한다.

## PaintBucketGenerator

`PaintBucketGenerator`는 물감통 오브젝트의 생성만 담당하는 `MonoBehaviour`다.

- 물감통 View 프리팹 참조
- 생성된 물감통을 배치할 부모 Transform 참조
- 공통 `PaintBucketVisualData` 참조
- 전달받은 목록과 같은 순서로 물감통 프리팹 생성
- 종류별 Sprite 조회 후 View에 범위와 Sprite 전달
- 생성된 View 배열 반환

Generator는 생성된 목록을 필드에 보관하거나 제거하지 않는다. 생성 이후의 소유권과 생명주기 관리는 Controller에 있다. 따라서 Generator의 역할은 입력 데이터를 런타임 UI 오브젝트로 만드는 데서 끝난다.

현재 생성 도중 데이터나 Sprite 오류가 발생하면 앞서 생성된 View가 Controller에 반환되지 못하고 남을 수 있다. 에셋 검증이 필요해지는 단계에서는 생성 전에 전체 입력과 Sprite를 검증하거나 실패 시 부분 생성물을 정리해야 한다.

## PaintBucketView

`PaintBucketView`는 생성된 물감통 한 개의 시각적 표현과 클릭 입력을 담당하는 `MonoBehaviour`다.

- UI Button 클릭 감지
- 물감통 Sprite 표시
- 확산 범위 숫자 표시
- 클릭 시 자신을 `Clicked` 이벤트로 전달
- 선택 상태 표현 API 제공
- 소모 및 복원 상태 표현

현재 `SetSelected`는 향후 강조 연출을 위한 빈 확장 지점이다. `SetConsumed(true)`는 Button을 비활성화하고 GameObject를 숨기며, `SetConsumed(false)`는 되돌리기 시 다시 활성화한다.

View는 물감 종류를 판정하거나 소모 가능 여부를 결정하지 않는다. Generator가 선택한 Sprite와 표시할 범위를 받고, Controller가 결정한 선택·소모 상태를 표현한다.

View는 `PaintBucket` 원본 데이터를 보관하지 않는다. 버튼 클릭 시 자신을 `Clicked` 이벤트로 바로 전달하고, Controller가 View 조회 테이블을 통해 대응하는 데이터와 런타임 상태를 찾는다.

## PaintBucketController

`PaintBucketController`는 생성된 물감통들의 인게임 상호작용과 런타임 상태를 관리한다.

### 초기화와 생명주기

`Initialize(IReadOnlyList<PaintBucket>)`는 다음 순서로 동작한다.

1. 기존 View의 클릭 이벤트를 해제하고 GameObject를 제거한다.
2. 기존 목록, View 조회 테이블, 선택 상태를 비운다.
3. Generator로 새 View 목록을 만든다.
4. 데이터와 View를 생성 순서 ID로 묶어 `BucketEntry`에 저장한다.
5. 각 View의 클릭 이벤트를 구독한다.
6. `GridView.CellClicked`를 해제 후 재구독해 중복 구독을 방지한다.

Controller가 파괴될 때는 GridView와 View 이벤트를 해제하고 생성한 물감통 오브젝트를 제거한다.

### BucketEntry

`BucketEntry`는 Controller 내부에서 물감통 하나를 안전하게 관리하기 위한 private 항목이다.

```text
BucketEntry
├─ Id: 현재 스테이지 안에서의 개별 식별자
├─ Data: PaintType과 Range 원본 사양
├─ View: 대응하는 화면 오브젝트
└─ IsConsumed: 런타임 소모 상태
```

같은 색상과 범위를 가진 물감통이 여러 개일 수 있으므로 사양만으로 개체를 식별할 수 없다. 생성 순서 ID를 사용하면 커맨드가 정확히 어느 물감통을 소모하고 복원할지 기록할 수 있다. ID는 스테이지를 다시 초기화하면 재할당되므로 기존 커맨드 스택도 함께 비워야 한다.

`BucketEntry.SetConsumed`는 논리적인 `IsConsumed`와 View의 표시를 함께 변경해 둘이 어긋나는 것을 막는다. 별도의 공개 `PaintBucketInstance`나 `PaintBucketRuntimeState`는 두지 않고 이 작은 런타임 상태를 Controller 내부 구현으로 유지한다.

### 선택 처리

View 클릭을 받으면 Dictionary로 대응하는 Entry를 찾는다.

- 소모된 물감통 클릭은 무시한다.
- 선택된 물감통을 다시 클릭하면 선택을 해제한다.
- 다른 물감통을 클릭하면 이전 선택을 해제하고 새 물감통을 선택한다.
- Controller가 비활성화된 동안 들어온 이벤트는 `isActiveAndEnabled` 검사로 무시한다.

### 셀 클릭과 사용 요청

Controller는 개별 `CellView`를 직접 구독하지 않고 `GridView.CellClicked` 이벤트 하나만 구독한다. 선택된 물감통이 있고 아직 소모되지 않았다면 다음 정보를 `BucketUseRequested`로 전달한다.

- 물감통 ID
- `PaintBucket` 사양
- 클릭한 셀의 논리 좌표

이 이벤트는 요청일 뿐이다. 현재 Controller는 셀의 확산 범위를 계산하거나 `GridState`를 변경하지 않으며 요청 직후 물감통을 자동 소모하지도 않는다. 향후 커맨드 실행이 성공했을 때 `Consume(id)`를 호출하고, Undo에서는 `Restore(id)`를 호출한다.

## 커맨드와 되돌리기 확장 경계

예정된 물감 사용 커맨드는 최소한 다음 정보를 기억한다.

- 사용한 물감통 ID
- 적용 중심 셀 좌표
- 영향을 받은 셀들의 변경 전 상태

실행 시 확산 좌표를 계산하고 셀 상태를 변경한 뒤 `PaintBucketController.Consume(id)`를 호출한다. Undo 시 셀들의 이전 상태를 복원하고 `PaintBucketController.Restore(id)`를 호출한다. 여러 커맨드를 Stack에 쌓으면 전체 격자 스냅샷을 행동마다 저장하지 않고도 여러 단계 되돌리기를 구현할 수 있다.

## 현재 구현 범위와 남은 작업

현재 완료된 구조는 데이터 정의, 종류별 Sprite 조회, View 생성, 개별 ID 관리, 선택 로직, 소모·복원 API, GridView 셀 클릭 중계 연결이다.

다음 작업은 아직 구현되지 않았다.

- 실제 선택 강조 표현
- 벽을 반영한 확산 범위 계산
- `PaintState` 적용과 CellView 색상 동기화
- 물감 사용 커맨드 및 Undo Stack
- 스테이지 초기화 시 커맨드 Stack 정리
- 생성 전 데이터·Sprite 검증 또는 부분 생성 실패 정리
- 선택·소모 전용 Sprite가 필요할 경우 VisualData API 확장
